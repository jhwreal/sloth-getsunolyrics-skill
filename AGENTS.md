# AGENTS.md

## 项目定位

本仓库维护和开发开源项目 `sloth-getsunolyrics-skill`。

该 Skill 接收歌曲名称，用 Computer Use 像人一样操作用户已登录的 Suno：复制歌词、下载带歌词画面的 MP4，并下载 Suno 分离的 Lead Vocal，最终生成可靠的毫秒级起点 CSV。Suno 页面歌词是正文来源，MP4 高亮是顺序与粗时间依据，歌词提示的 whisper.cpp DTW 是内容感知的逐字时间依据，人声活动用于拒绝不可能的 DTW 回跳。

## 核心原则

1. **先核对文本，再定时间。** 从 Suno 可见歌词区复制暂定正文；先用 MP4 核实 Suno 实际生成的文字，一致时继续原流程，真实冲突时列出差异并由用户决定。不得用未经画面核对的 OCR 或语音识别擅自改写歌词。
2. **保留原意与原貌。** 不随意润色、翻译、简繁转换或替换重复句。Suno 歌词中的语言、标点、大小写、重复和分行应尽可能保留。
3. **校准必须有依据。** 时间修正应由内容感知的 DTW token、人声 onset、MP4 顺序窗口和明确算法共同支撑，不使用无依据的固定偏移掩盖问题。
4. **结果可追溯。** 保存或记录输入、关键中间产物、参数、置信度、警告和校准方式，使结果能够复现和复核。
5. **不确定性要显式表达。** 遮挡、模糊、和声、拖音、重叠演唱或分离伪影导致的不确定项，不可静默猜测；应标记低置信度或生成待人工复核信息。
6. **原始媒体只读。** 不覆盖用户提供的 MP4、音频或歌词文件。生成物写入独立输出目录。

## 标准处理流程

实现和修改功能时，应围绕以下流水线组织：

1. **Computer Use 取材与输入检查**
   - 用户只需提供歌曲名称并在 Chrome 登录 Suno。所有 Suno 操作用 Computer Use 模拟人的可见点击与复制。
   - 搜索并核对标题、作者、版本、日期、封面、时长和 UUID，不把推荐歌曲、底部播放器或同名版本误当成目标歌曲。
   - 从可见歌词区复制无时间戳歌词，从可见菜单下载 Video，并通过 Get Stems 下载完整 Lead Vocal。
   - 若最终操作显示扣积分、付费或升级，执行前展示具体费用并确认。
   - 下载后记录歌词、MP4 和人声哈希与时长，再复制到当前歌词处理工作目录。
   - 验证 MP4 是否存在且可解码。
   - 读取时长、帧率、分辨率、音轨、采样率等元数据。
   - 对无歌词画面、无音轨、可变帧率或损坏媒体给出清晰错误或警告。

2. **歌词与画面提取**
   - 输入必须包含从 Suno 复制的无时间戳歌词；人工时间轴 CSV/TS 不得作为生成输入。
   - 按时间采样或根据画面变化定位歌词区域和切换时刻。
   - OCR 识别歌词文本，并合并跨帧重复结果。
   - 形成按演唱顺序排列的歌词行及其粗略显示区间。
   - 记录 OCR 置信度和存在歧义的候选文本。
   - 在按原歌词过滤 OCR 之前比较两套行序列，避免把 Suno 真正改写的歌词误删为噪声。
   - 高证据差异必须先对照 MP4 可见画面；OCR 误差自行排除，真实差异输出时间点和两版文字并暂停等待用户选择。
   - 未发现真实冲突时不得改变既有对齐流程。用户选择 MP4 版时保留原始复制文本并另存核对后的修订文本；用户选择复制版时记录显式冲突警告。

3. **音频与人声处理**
   - 默认使用 Suno Stem 页面得到的 Lead Vocal，不要求普通用户本地安装 Demucs。
   - 验证人声轨与 MP4 同源、同起点、时长一致，并记录来源和哈希。
   - 必要时统一采样率、声道和响度，但不得引入未记录的时间伸缩。

4. **时间校准**
   - 用规范歌词全文提示本地 whisper.cpp；固定 `-mc 0`，启用 DTW，并用 `-nfa` 关闭会禁用 DTW 的 Flash Attention。识别文本只能提供时间证据，不得改写规范歌词。
   - 按顺序对齐歌词组与 Whisper 句段；一个句段合并多行歌词时，后续歌词必须定位到句段内部的对应 token，不得共用整段起点。
   - `t_dtw` 若比所属 Whisper 句段起点早 500 ms 以上，应判定为模型回跳并拒绝；再以原句段起点附近的向前人声 onset 修正，同时记录警告。
   - 首行弱匹配若跳过了更早的 Whisper 片段，只能在其早期 DTW 与 Lead Vocal onset 互相确认时恢复；连续两行若得到相同 DTW 起点，应以 MP4 顺序锚点恢复前一条遗漏行，不得压成相邻 1 ms。
   - MP4 粗时间只作为顺序、搜索窗口和审计证据，不得套用无依据的固定偏移；段落歌词可能提前显示，画面切换也可能滞后于人声。
   - 将歌词行的开始时间对齐到可辨识的演唱起点；本项目不再推断结束时间。
   - 对前奏、间奏、尾奏、拖音、抢拍、叠唱、和声及重复歌词进行专门处理。
   - 校准后必须保持起点严格递增，避免负时间和倒序。

5. **结果生成与验证**
   - 以内部统一数据模型保存歌词行、时间、文本、来源和置信度。
   - 默认导出网易云样式的起点 CSV，同时生成 start-only LRC 和可追溯 JSON；不得生成依赖结束时间的 SRT/VTT。
   - 运行结构校验、起点单调性和媒体时长校验。
   - 完整包必须包含歌词比较报告，且不得处于 `decision_pending` 状态。
   - 为低置信度结果输出明确的人工复核提示。

6. **网页人工校对与定版**
   - 一首或多首歌曲完成盲生成与结构校验后，用 `scripts/review_timeline.py` 启动仅监听本机回环地址的两栏校对台。
   - 左栏只负责切换歌曲；右栏将播放器、逐句歌词、当前句时间写入/微调、整轨偏移、保存和定版合并在同一工作区。
   - 点击歌词必须同步跳播；“添加时间点”写入播放器当前整数毫秒。服务端必须拒绝负数、越界、重复、倒序和过期 revision。
   - 首次人工保存前将机器生成的 JSON/CSV/LRC/manifest 备份到 `review/original/`；人工修改保留自动起点与来源，重新导出并验证，不得破坏盲评证据。
   - 人工答案或网易云基准仍然只能在生成完成后评测，不能因网页校对功能而进入生产生成输入。

## 时间轴语义

- 内部计算统一使用整数毫秒，避免浮点累计误差。
- `start_ms` 表示该行实际开始演唱的时间，而不只是歌词首次出现在画面上的时间。
- 所有时间必须满足：
  - `0 <= start_ms < media_duration_ms`
  - 歌词行整体按 `start_ms` 严格递增排列

建议的内部歌词行结构：

```json
{
  "index": 1,
  "text": "歌词内容",
  "start_ms": 12340,
  "text_source": "suno_lyrics_confirmed_by_video",
  "timing_source": "whisper_dtw_token_start",
  "confidence": 0.94,
  "flags": []
}
```

## 工程约定

- 处理阶段应模块化，至少区分媒体读取、画面/OCR、人声分离、时间对齐、格式导出和质量验证。
- 外部工具和模型通过明确接口调用，避免把机器相关的绝对路径写死在代码中。
- 对 FFmpeg、Vision OCR、`whisper-cli`、DTW 模型和人声分离回退等依赖进行启动前检查，并提供可操作的缺失依赖提示。
- 默认行为应确定且可复现。随机算法固定种子；模型版本和参数写入元数据或日志。
- 日志不得泄露用户隐私、访问令牌或本机敏感路径。日志要能定位处理阶段、输入片段和失败原因。
- 长任务应报告阶段与进度，并支持失败后安全重试；已有有效中间产物可缓存，但缓存键必须包含源文件指纹和相关参数。
- 时间轴缓存键必须包含媒体/歌词哈希、OCR 语言、采样间隔、流水线指纹、Whisper 可执行文件哈希、模型哈希和 DTW 参数；原始 OCR 缓存必须先清除异源旧帧，禁止跨歌曲污染。
- 禁止在通用实现中加入歌曲标题、歌词句子或当前基准样本专用的过滤条件。
- 临时文件应集中管理并可清理。不要把大型媒体、中间音轨、模型权重或生成结果误提交到 Git。
- 新增技术栈、目录结构或命令后，同步更新本文件、`SKILL.md` 和 `README.md`，不要让维护约定落后于代码。

当前入口和依赖：

- `scripts/process_song.py`：端到端打包与断点续跑入口。
- `scripts/media_utils.py`：媒体流、时长与配对预检。
- `scripts/lyrics_conflict.py`：复制歌词与未过滤 MP4 OCR 行序列比较、差异定位及双语报告。
- `scripts/separate_vocals.py`：仅供开发者回退的 Demucs 本地人声分离。
- `scripts/extract_timeline.py`：Suno 歌词、macOS Vision OCR、Whisper DTW 与人声证据的流水线编排。
- `scripts/whisper_alignment.py`：本地 whisper.cpp DTW、歌词/句段顺序对齐、合并句段内 token 定位及回跳拒绝。
- `scripts/export_timeline.py`：JSON 转网易云四列起点 CSV 和 start-only LRC。
- `scripts/evaluate_timeline.py`：与人工核对 CSV/TypeScript 对照评测。
- `scripts/validate_package.py`：哈希、时长、歌词正文、起点单调性及所有导出正文/时间一致性验证。
- `scripts/review_timeline.py`：本地多歌曲时间轴校对服务，提供播放跳转、逐句写点、整体偏移、原始结果备份与定版保存。
- `assets/review-ui/`：无外部依赖的两栏校对台前端和中文使用教程图。
- `requirements-demucs.txt`：仅供开发者离线回退的人声分离依赖，不是普通用户前置要求。

长任务失败后优先使用 `--resume`，不得在已有可验证人声轨时无意义地重复模型推理。

## 测试与验收

每次修改应采用与风险相称的验证。至少覆盖：

- 无音轨、无歌词画面、空歌词和损坏 MP4。
- 中英文、混合语言、重复句、双行字幕和歌词快速切换。
- 前奏/间奏、拖音、静音、和声、叠唱和人声分离伪影。
- 固定帧率与可变帧率视频，以及不同采样率音频。
- OCR 误识别、歌词画面提前出现或延迟消失的情况。
- 页面歌词与 MP4 完全一致、单词/汉字被 Suno 改写、视频新增歌词行、片头标题、OCR 弱证据错字和用户三种冲突决策。
- 时间戳单调性、合法范围、导出格式解析和往返一致性。
- 多歌曲切换、媒体 Range 请求、点击跳播、逐句写点、整轨偏移、并发 revision、机器结果备份和人工定版状态。

测试分层建议：

1. 纯函数和格式转换使用单元测试。
2. 各处理阶段使用小型固定样本进行集成测试。
3. 端到端测试验证从 MP4 到时间轴文件的完整流程。
4. 对关键样本记录可接受误差；当前维护的精度回归要求每一行（不是只看中位数或 p95）绝对起点误差不超过 500 ms，并同时报告最大值、中位数、p95 与有符号偏差。

涉及时间对齐算法的改动，必须在生成完成后才读取独立人工答案，报告逐行起点绝对误差、最大值、匹配率或低置信度比例。人工答案不得进入 prompt、缓存键、候选选择或任何生产输入；不能只凭肉眼判断“更准”。

## Agent 工作方式

在本仓库工作的 Agent 应：

1. 修改前先检查 `SKILL.md`、依赖清单、测试和相关实现；若这些文件尚不存在，先理解当前仓库状态，不虚构已有接口或命令。
2. 优先定位根因，保持改动范围聚焦，不顺手重构无关代码。
3. 保留用户已有改动，不覆盖、不回滚无关文件。
4. 新增行为时同时补充测试、错误处理和必要文档。
5. 不在没有代表性样本或指标的情况下宣称对齐精度提升。
6. 无法运行依赖模型或端到端验证时，明确说明未验证的部分、原因和用户可执行的验证方法。
7. 完成后简要交代修改内容、验证结果、生成文件位置及剩余风险。

## 开源与隐私

- 提交前确认新依赖、模型和测试素材的许可证允许开源及目标用途。
- 不提交来源不明的歌曲、完整商业音频、视频或受版权保护的歌词语料。
- 测试夹具优先使用自制、授权、公共领域或程序生成的短样本。
- 不上传用户媒体到第三方服务，除非功能明确要求且用户已知情同意。
- 若使用云端 OCR、转写或分离服务，应在文档中说明会发送的数据、保留策略、费用和可选的本地替代方案。
